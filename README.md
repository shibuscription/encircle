# あっちむいて包囲

「あっちむいてホイ」と「包囲」を掛け合わせた、1人用のカジュアル・パズルゲームのプロトタイプ案です。  
スマホブラウザを主な対象にしつつ、PC ブラウザでも遊べる構成を想定しています。

このリポジトリでは、まずはゲームの核となる体験を素早く形にすることを優先し、遊べるプロトタイプを早期に作ることを目的とします。ゲームバランスや難易度は、実際に触りながら段階的に調整していく想定です。

## アプリ概要

- アプリ名: あっちむいて包囲
- ディレクトリ名: `encircle`
- ジャンル: 1人用カジュアル・パズルゲーム
- 想定プラットフォーム: スマホブラウザ中心、PC ブラウザ対応も視野
- 開発方針: まずは最小構成で遊べるプロトタイプを作り、その後に手触りとゲームバランスを詰める

## ゲームコンセプト

毎ターン、プレイヤーは上下左右のどこへ進むかを選び、同時に敵も方向を選びます。  
敵は移動せず、プレイヤーの進路を少しずつ塞いでいく障害物として盤面に増えていきます。

このゲームの魅力として想定しているのは、以下のような感覚です。

- その場その場で安全な方向を選ぶ判断
- 選択肢が徐々に奪われていく圧迫感
- まだ逃げ道があるうちに出口へ向かう緊張感
- 「包囲されていく感覚」そのものを楽しむ短時間プレイ

## 現時点のコアルール

以下は、まずプロトタイプに反映したい仮仕様です。

- 盤面は `7×7` のマス目
- プレイヤーは中央マスから開始
- 四隅の 4 マスが出口
- 毎ターン、「あっちむいて包囲！」としてプレイヤーと敵が同時に上下左右のいずれかを選ぶ
- プレイヤーは、盤外方向および隣接マスに敵がいる方向は選べない
- UI では、選べない方向ボタンは無効化する想定
- プレイヤーと敵が違う方向を選んだ場合、プレイヤーはその方向へ 1 マス移動する
- プレイヤーと敵が同じ方向を選んだ場合、プレイヤーが選んだ方向の隣接マスに敵が 1 体出現する
- 敵は動かない障害物であり、攻撃や追跡はしない
- プレイヤーは敵のいるマスには進めない
- 出口マスには敵を置けない
- プレイヤーが出口方向を選んで出口マスへ到達できる状態なら、そのターンは即クリア扱いでよい
- 敗北判定は、現在位置から 4 つの出口のどれにも到達できなくなった時点とする
- 敗北判定の実装は、経路探索ベースで行う想定

## 勝利条件・敗北条件

### 勝利条件

- プレイヤーがいずれかの出口マスへ到達する
- 特に、選択した進行方向の 1 マス先が出口マスで到達可能なら、そのターンで即クリア扱いとする

### 敗北条件

- 現在位置から、4 つの出口のどれにも到達できない状態になる
- 判定は「今いる位置から出口までの到達可能経路が 1 つもない」ことを基準にする

単に周囲 1 マスが埋まったかどうかではなく、盤面全体で逃げ道が残っているかを判定する想定です。

## 盤面と UI の基本イメージ

初期段階では、見た目よりもルール確認と手触り検証を優先します。

- 画面中央に 7×7 の盤面を表示
- マスの種類は、通常マス / プレイヤー / 敵 / 出口 が判別できる程度のシンプルな表現から開始
- 画面下部または親指で押しやすい位置に上下左右の方向ボタンを配置
- 選択不能な方向ボタンは無効化して、押せないことが視覚的にわかるようにする
- スマホ縦持ちでの見やすさと操作しやすさを優先
- PC 表示では盤面と操作ボタンが崩れないように最低限レスポンシブ対応する

将来的には演出やアニメーションを加える余地はありますが、初期プロトタイプでは必須にしません。

## 今後調整予定の項目

ゲームバランスはまだ仮であり、以下は試遊しながら調整していく前提です。

- 敵の方向選択ロジック
- 盤面サイズが `7×7` で適切かどうか
- 開始位置が中央固定でよいか
- 出口の数と配置
- 難易度の立ち上がり方
- 敵出現テンポによる圧迫感
- 1 プレイの所要時間
- UI の視認性と押しやすさ

まずは厳密な最適化よりも、「包囲されていく感覚」が成立しているかを確認することを優先します。

## 技術構成

現時点での初期想定です。必要に応じて今後見直します。

- ビルド / 開発環境: Vite
- UI: React
- 言語: TypeScript
- 状態管理: まずは React Hooks 中心で構成
- スタイリング: シンプルな CSS または CSS Modules から開始
- ロジック:
  - 盤面管理
  - ターン進行管理
  - 方向選択の可否判定
  - 勝敗判定
- 経路探索: シンプルな BFS / DFS 系ロジックを想定

## 開発メモ

- 最初から作り込みすぎず、1 プレイが成立する最小構成を優先する
- ロジックと表示はなるべく分離し、ルール調整しやすい形を目指す
- 敵は「動くキャラクター」ではなく「増えていく障害物」として扱う
- プレイヤーにとって選べない方向は、ロジック上だけでなく UI 上でも明確にする
- 敗北判定は重要な体験部分なので、経路探索の仕様は早めに固める
- 数値やルールは固定値として埋め込むより、後で調整しやすい定数化を意識する

## 今後の実装ステップ案

1. Vite + React + TypeScript でプロジェクトをセットアップする
2. 7×7 盤面とプレイヤー初期位置、出口マスを表示する
3. 上下左右ボタンと、選択不能方向の無効化ロジックを実装する
4. プレイヤー移動と敵出現の同時判定ルールを実装する
5. 敵マス・出口マスを含めた盤面更新処理を整える
6. 勝利条件と、経路探索ベースの敗北条件を実装する
7. スマホ画面を優先したレイアウト調整を行う
8. 複数回試遊して、テンポや難易度の仮調整を行う
9. 必要に応じて演出、効果表示、リトライ導線などを追加検討する
